Omokai

status: Work in progress

Solo developer · technical assignment

Omokai closes the loop from spoken intent to physical robot motion. A natural-language instruction is parsed by an LLM into a structured mission, checked against a deterministic safety validator, and executed as a real Nav2 navigation task inside Gazebo. The goal wasn't just to make an LLM output JSON, it was to build the guardrails that make it safe to act on.

KEY DECISIONS —Chose Groq (llama3-8b-8192) over local inference. Ollama technically worked but was unusably slow on CPU-only hardware, and other hosted providers hit access or deprecation issues. Groq's free tier gave the best latency-to-reliability tradeoff, the priority was a model that responds consistently, not the most capable one available. Put a hard validation layer between the LLM and the executor. The LLM is treated as untrusted input. Nothing reaches Nav2 without first passing schema checks and safety bounds, this was a deliberate design choice, not an afterthought, because an LLM hallucinating a bad waypoint should fail loudly before it ever reaches a motor command. Diagnosed and fixed a Nav2 activation hang at its root cause. The stall traced back to a slam_toolbox lifecycle dependency blocking Nav2's localizer check. Rather than papering over it with retries or timeouts, the check itself was bypassed once the actual dependency was identified, fixing the cause, not the symptom.

XLR8

status: voice pipeline complete

Solo developer · voice-driven robotics assistant

A modular voice-assistant system built on ROS 2, where independent nodes handle speech recognition, intent classification, LLM-based reasoning, and speech synthesis, communicating entirely through ROS topics, services, and actions. Designed as the interaction layer for a robotics platform, with a defined interface for downstream robot-action execution.

key decision — implemented speech output as a cancellable ROS 2 action rather than a fire-and-forget service, enabling real-time barge-in: the system stops talking the instant the user starts speaking again.

A modular, cost-efficient autonomous drone system for disaster response. The UAS autonomously scans a user-defined area, detects disaster zones with a YOLOv8-based vision model, and performs payload drops without ground control, fusing LiDAR, PX4 telemetry, and GPS onboard for precise delivery. Quadruped Robot

status: earlier project

Solo developer · personal project

A 12-DOF quadruped robot built from the ground up on an Arduino Nano and twelve SG90 servo motors, with coordinated gait control and motion sequencing for walking and expressive movements.

key decision — used predefined gait sequences with synchronized joint trajectories, prioritizing repeatable locomotion over computationally intensive approaches like inverse kinematics, to stay within the Arduino Nano's headroom.
